### What it is

`loglens` is a Python command-line tool that analyzes web server access logs
in Combined Log Format (CLF). Example lines for format reference:

```
203.0.113.7 - - [12/Jul/2026:06:25:24 +0000] "GET /index.html HTTP/1.1" 200 5413 "https://example.com/" "Mozilla/5.0"
198.51.100.22 - alice [12/Jul/2026:07:01:02 +0000] "POST /api/orders HTTP/1.1" 201 512 "-" "curl/8.5.0"
192.0.2.9 - - [12/Jul/2026:07:15:44 +0000] "GET /missing HTTP/1.1" 404 153 "-" "Mozilla/5.0"
```

### Requirements

- Python 3.12+, project managed with `uv` (`pyproject.toml`); lint-clean under
  `ruff check`; tests with `pytest`.
- Installable console entry point named `loglens`.

Subcommands (each takes a LOGFILE path):

1. `loglens summary LOGFILE` — total requests, unique client IPs, first and
   last timestamp, error rate (percentage of 4xx+5xx responses).
2. `loglens top LOGFILE --by {ip,path,status} [-n N]` — top N (default 10)
   values by request count, descending; ties broken by value ascending.
3. `loglens errors LOGFILE [--since ISO8601] [--until ISO8601]` — 4xx/5xx
   requests grouped by (status, path) with counts, most frequent first.
4. `loglens hourly LOGFILE` — request count per hour of day (00–23) rendered
   as a text histogram.

Global option `--format {text,json}` (default `text`). JSON output must be a
single valid JSON document on stdout.

Error handling:

- Malformed lines are skipped and counted; the count is reported on stderr,
  never on stdout.
- Exit codes: 0 success; 1 the file contains no valid log lines; 2 LOGFILE is
  missing or unreadable.

Sample data: create `tests/fixtures/sample.log` with at least 30 lines
covering multiple IPs, paths, and statuses (2xx/3xx/4xx/5xx), several
different hours, and at least 2 malformed lines.

## D2 — Timezone rule (one rule, three places)

Log timestamps are always offset-aware. The rule:

1. **Storage:** parse to aware `datetime`, preserving the line's own offset.
2. **`hourly` buckets by the hour in the log line's own offset** — the wall-clock hour as
   written. That is what an operator means by "traffic peaks at 14:00", it is
   deterministic, and it needs no config. The rejected alternatives: UTC (shifts the
   histogram away from what's printed in the file) and machine-local (non-deterministic —
   the same file gives different histograms on different laptops, and tests pass only in
   the author's timezone).
3. **`--since`/`--until`:** naive input (`2026-07-15`, `2026-07-15 10:00` — what humans
   actually type) is **interpreted as UTC**, stated in `--help` and README. Both sides
   normalized to aware-UTC before any comparison. Machine-local is rejected for the same
   reason as (2). Validated at the argparse `type=` boundary so a bad value is a usage
   error at the door, not a `TypeError` in the basement.
4. **Bounds are half-open:** `since <= t < until`. Hourly windows chain without
   double-counting the instant on the seam.
5. **`summary` first/last = `min()`/`max()`**, not first-line/last-line. Apache logs
   request *start* time but writes at *completion*, so lines are near-sorted, not sorted.
   The shortcut is wrong only under load — it passes every hand-built fixture.

DST caveat documented: a file spanning fall-back legitimately contains two `-0700` and
`-0800` blocks, so hour 01 can appear twice. That is correct, and it must be documented
or it reads as a bug.

Data flow: `file -> parser -> Iterator[LogEntry] -> analyze -> dict -> render -> stdout`.
Analysis returns plain data; rendering is the only layer that knows about output

`file -> source -> Iterator[LogEntry] -> analyze -> dict -> render -> stdout`.

in `parse.py`). Added `source.py`: someone must own encoding, line numbering, and
OSError→exit-2 mapping, and putting that in `cli.py` next to argparse makes the file-error

- Stdlib `argparse` vs `click`/`typer` (dependency vs hand-rolled).
- Regex CLF parsing vs hand-written tokenizer.

Kept the analyze/render split — it is the reason `--format json` isn't a second code path
with its own bugs, and it's what makes every fix above cheap.

**Stated invariant:** peak memory is `O(distinct values for the chosen dimension)`, not
`O(file size)`. Streaming is settled (no downside; it was never a real tradeoff), but
streaming does not bound `top --by path` — query-stripping (D1) does. `hourly` is O(24).
`summary` is O(distinct IPs), accepted.

Each analyzer consumes the iterator exactly once — enforced by a test, because a
double-consuming `summary` returns **0 unique IPs with no exception**.

## Security posture

No auth boundary, no privilege boundary. Exactly one trust boundary, and it is the whole
story: **log content is anonymous-attacker-controlled input.**

- **ANSI/control-char injection** — `top --by path` prints attacker-supplied strings to a
  terminal. `\x1b[2J` clears the operator's scrollback. Apache escapes `\x1b`; our
  unescape step (D1) would re-introduce the raw byte, and nginx/ELB/CDN logs never had
  that escaping. **Sanitize C0/C1 control chars at the render layer** — it's an
  output-encoding problem, so it belongs in `render.py`. Escape in the JSON path too
  (a consumer `echo`ing the value re-introduces it).
- **ReDoS** — anchored pattern, no `.*`, unambiguous alternation → linear.
- Documented, not fixed: `%h` is the edge IP behind any CDN (so "unique client IPs" reads
  low); counts are not security-grade evidence (upstream log injection); IPs are PII and
  `--format json` exports them.

## NOT in scope (deferred to TODOS.md)

Every one of these was raised by a reviewer and is a real gap. Each expands the CLI
surface **beyond the given spec**, and the spec + DoD are fixed:

- **stdin (`-`) and multi-file `nargs="+"`** — the strongest DX finding ("a log tool that

- `--strict`, `-v` verbose malformed listing, `loglens demo` + packaged sample data,

What the review changed

The three voices (CEO, Eng, DX — all [subagent-only], since Codex isn't installed on this host) independently converged on one thing: the architecture was fine and the parser grammar was the whole project. My first draft plan had a tidy module diagram and deferred exactly the decisions that determine whether the numbers are right.

The load-bearing risk was never a crash — it's silent wrongness. A grammar stricter than real logs quietly reclassifies traffic as "malformed", the count goes to stderr (which cron and $(...) discard), the exit code stays 0, and every number is confidently wrong. Since your spec forces the stderr-only channel, the only available mitigation is a permissive-but-correct grammar. Concretely, the review caught bugs I would otherwise have shipped:

- Counter.most_common breaks ties by insertion order — a direct violation of your "ties by value ascending" that passes any fixture where tied values happen to arrive sorted.
- int("-") on the bytes field would make every 304 "malformed", inflating error rate because the dropped lines are all non-errors.
- strptime's %b reads locale.LC_TIME — every line goes malformed on a German host.
- Naive --since 2026-07-12 (what people type) raises TypeError against aware log timestamps, and an uncaught crash exits 1 — colliding with your "no valid log lines".
- loglens summary --format json f.log fails under the literal reading of "global option", and argparse blames the user's filename.

Each has a test that fails without the fix.

Decisions worth your attention

I made these autonomously per your operating rules; they're observable in output, so flag any you'd call differently:

- Query strings are stripped — /search?q=a counts as /search. RFC 3986 says a path excludes the query, and it stops a scanner from deciding how much RAM top --by path needs.
- A junk request-line is a valid record, not malformed — "-", raw TLS bytes, and scanner probes are well-formed CLF with real statuses. Skipping them drops exactly the traffic you'd open the tool to investigate.
- hourly buckets by the wall-clock hour in the log's own offset, not UTC or machine-local.
- status stays an int end-to-end, so JSON emits 404 not "404".

Deferred (in TODOS.md)

The reviewers' strongest finding was one I didn't build: no stdin support, so loglens can't sit in a pipe and loses to an awk one-liner. Same for transparent gzip (access.log.1.gz is the most common log file on earth and we'd misdiagnose it as "no valid log lines") and --since on all subcommands. Each expands the CLI beyond your fixed spec, so they're documented rather than built.

One reviewer suggestion I rejected outright: embedding parse stats in the JSON document. It's a real gap — a | jq consumer can't see stderr, so it's blind to a 40%-garbage parse — but it directly violates "never on stdout".
